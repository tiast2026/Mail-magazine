# CLAUDE.md — メルマガ管理システム運用ガイド

このリポジトリは楽天店舗向けメルマガ制作・配信実績管理システムです。
このファイルは Claude Code が新セッション開始時に参照する想定で書かれています。

## システム概要

- **AI 生成役**: Claude Code（このセッション）— 楽天 RMS API で商品情報取得 → テンプレに流し込み → outputs.json に追記 → push
- **Web UI**: Vercel 静的ホスティング — 閲覧・プレビュー・HTML コピー・実績入力
- **データ**: リポジトリ内 JSON で一元管理（push が即 Web 反映）
- API 課金は発生しない設計（生成は CLI 側で完結）

## 主要ファイル

| パス | 役割 |
|------|------|
| `frontend/data/brands.json` | ブランド一覧 |
| `frontend/data/brands/<brandId>/config.json` | ブランド情報（色・ロゴ・固定URL・アイデンティティ） |
| `frontend/data/brands/<brandId>/templates.json` | ブランド別テンプレート（A/B/C/D） |
| `frontend/data/brands/<brandId>/outputs.json` | 生成済みメルマガと配信実績 |
| `frontend/data/events.json` | 楽天キャンペーン日程カレンダー（マラソン・SS等） |
| `frontend/app/api/rakuten/[brandId]/[manageNumber]/route.ts` | 楽天 RMS API ルート |
| `frontend/lib/data.ts` | データ取得 + `applyBrandToHtml()` 色置換ヘルパー |
| `frontend/lib/events.ts` | キャンペーンイベント定義（marathon, supersale 等） |
| `scripts/fetch-rakuten-public.mjs` | 公開ページから OGP 取得（フォールバック） |

## 楽天 RMS API の使い方

**認証情報は Vercel 環境変数で管理**しています。Claude Code は
デプロイ済みの API ルート経由で商品情報を取得します。

### API エンドポイント

```
GET https://mail-magazine.vercel.app/api/rakuten/<brandId>/<品番>
```

例:
```bash
curl -s https://mail-magazine.vercel.app/api/rakuten/noahl/nlwp315-2505
```

レスポンス JSON:
- `manageNumber`: 商品管理番号
- `name`: 商品名
- `tagline`: タグライン
- `url`: 商品ページURL
- `mainImage`: 第一画像 URL（fullUrl）
- `images[]`: 全画像（location, fullUrl, alt）
- `variants`: カラー別 variants（images, standardPrice）

### 環境変数の命名規則

ブランド固有：
- `RAKUTEN_<BRANDID大文字>_SERVICE_SECRET`
- `RAKUTEN_<BRANDID大文字>_LICENSE_KEY`
- `RAKUTEN_<BRANDID大文字>_SHOP_URL`

例: NOAHL なら `RAKUTEN_NOAHL_SERVICE_SECRET`

旧命名（プレフィックスなし）はフォールバック対応。

### メルマガ生成時のフロー

ユーザーから「品番〇〇でメルマガ作って」と依頼されたら：

1. 現在編集中ブランド（通常は default = noahl）を確認
2. `curl https://mail-magazine.vercel.app/api/rakuten/<brandId>/<品番>` で商品情報取得
3. テンプレートを選定（A/B/C/D）
4. テンプレートに変数を埋めて HTML 生成（{{COLOR_*}} 等は applyBrandToHtml で実行時置換）
5. `frontend/data/brands/<brandId>/outputs.json` に追記
6. commit & push → Vercel 自動デプロイ → Web で確認可能

### イベント・配信日付情報

メルマガ生成時に以下も指定する：
- `event.type`: marathon / supersale / blackfriday / yearend / newyear /
  newcollection / preorder / restock / review / regular / custom
- `event.name`: 表示名（例: "2026年5月お買い物マラソン"）
- `event.startDate` / `event.endDate`: イベント期間
- `scheduledAt`: 配信予定日時
- `sentAt`: 実配信日時（後から記録）

ユーザーから「マラソン向けに作って」「スーパーSALE で送る」と言われたら、
適切な `event.type` を設定する。日時はユーザーから聞くか、現在時刻ベースで仮設定。

## セッション開始時の対応（必読・提案型）

ユーザーから来る最初の依頼は **「品番〇〇でメルマガ作って」** が典型。
**まず品番から商品を取得してから提案する** スタイルで動く：

### メルマガ生成の必須4項目

メルマガを push する前に **必ず以下4つが揃っていること** を確認する：

| # | 項目 | 取得元 |
|---|------|--------|
| 1 | **品番**（manageNumber） | ユーザーが最初に提示（必須） |
| 2 | **割引率**（クーポン適用後 % ／元値・割引後価格） | ユーザーが最初に提示（必須） |
| 3 | **件名**（subject） | Claude が「件名作成ルール」に沿って 1〜2 案提示 → ユーザー確認 |
| 4 | **配信日時**（scheduledAt） | Claude が直近イベントから提案 → ユーザー確認 |

ユーザーは通常 **1（品番）と 2（割引率）** だけを最初に渡してくる。
**3（件名）と 4（配信日時）は Claude 側で必ず提案して確認を取る**。確認なしで push しない。

### 流れ（商品取得 → 分析 → 件名・配信日提案 → 確認 → 生成）

#### Step 1: 品番から商品情報を取得（即時）

```bash
curl -s https://mail-magazine.vercel.app/api/rakuten/<brandId>/<品番>
```

取得した JSON から把握すべき要素：
- **商品名・タグライン** → 訴求軸のヒント
- **カテゴリ・素材** → 季節性・ターゲット推定
- **価格帯** → 単品プッシュ向きかセール抱き合わせ向きか
- **カラー数** → 1色なら単品プッシュ、2色以上ならカラー見本セクションが活きる
- **画像枚数** → 着画/詳細カットが豊富ならストーリー型 D が使える

#### Step 2: 直近の楽天イベントを `frontend/data/events.json` から確認

ファイルを読んで、今日から先 1〜2 ヶ月の以下イベントを把握：
- お買い物マラソン（毎月複数回開催）
- 楽天スーパーSALE（3, 6, 9, 12 月）
- ブラックフライデー、大感謝祭、初売り
- ワンダフルデー（毎月1日）
- 5と0のつく日

各イベントには **`announcementDate`（告知解禁日時）** がある。
**この日時より前にメルマガでイベントを告知することはできない**ので、配信日提案時は必ず確認：

- ✅ 配信は `announcementDate` 以降
- ✅ 開始2日前 〜 開始当日午前 が定番（盛り上げ）
- ❌ 告知解禁前の配信 → 楽天規約違反

季節イベントも考慮：母の日 (5月第2日曜)、父の日 (6月第3日曜)、敬老の日、ハロウィン、X'mas など。

#### Step 3: 過去配信実績を確認

`frontend/data/brands/<brandId>/outputs.json` を読んで：
- 同じ商品の過去配信があるか（重複回避 or 訴求軸切替）
- 似た商品 / カテゴリの反応データ
- 直近で使ったテンプレ（連続使用は避けて変化を出す）

#### Step 4: 提案を提示する（メイン1案 + 代替2案）

質問だけ並べるのではなく、**最も可能性が高い案を1つ提示してから、選択肢を3つ程度提案する**：

> 品番 nlwp315-2505 取得しました：
> - 商品: クロシェ編みドッキングワンピース（楊柳素材、ロング、2色展開）
> - 価格: 8,980円 → 5,388円（40%OFFクーポン適用）
> - 画像: モデル着画 9枚（イボリー）+ 8枚（ブラック）+ 詳細カット
>
> 直近のイベント:
> - 5/9（土）20:00 お買い物マラソン開始（GW明け、ジャンルSALE 同時）
> - 5/10（日）母の日
>
> **おすすめ案: 5/7（木）配信「母の日 × マラソン直前」**
> → テンプレ I（ギフト訴求型）で価格帯別ギフト提案、ナチュラル系ワンピは贈り物としても筋が良い
>
> 他の選択肢:
> - 5/9（土）配信「マラソン開始時の純粋セール告知」→ テンプレB（セール告知型）
> - 5/12（火）配信「楊柳素材ストーリー」→ テンプレD（ナチュラル読み物型）、ブランド世界観訴求
>
> どの方向で進めますか？商品追加が必要ならその品番も教えてください。

#### Step 5: 件名と配信日を必ず確認する（最重要）

ユーザーから渡される必須項目は通常 **品番＋割引率** のみ。
残り 2 項目（件名・配信日時）は **Claude が必ず提案 → 確認** を取る。

**件名（subject）**: 「件名作成ルール」セクションに沿って 1〜2 案を提示する。
例:
> 件名案、どちらが良いですか？
> A:【50%OFFクーポン】楊柳のうたう夏 ＊ 5,388円で迎える、私の主役ワンピース
> B:【お買い物マラソン】今だけ40%OFF ＊ 楊柳のクロシェ編みワンピース

**配信日時（scheduledAt）**: 直近のイベント・告知解禁日・通常運用時間（火水20-22時 or 土20:00 等）を踏まえて提案。
例:
> 配信日時、5/7（木）20:00 でいかがですか？
> （理由: マラソン告知解禁の 5/7 10:00 以降で、平日夕方の定番枠）

**他に追加で確認すべきもの（あれば）**:
- 商品数が不足 → 「テンプレB なら3点必要、追加品番ありますか？」
- バナー画像が必要なテンプレ（B/C/I）→ 「バナーURLありますか？なければ商品画像で代用」

#### Step 6: 4項目揃ったら即生成 → push

**4項目（品番・割引率・件名・配信日時）が揃って初めて push** する。
件名 or 配信日時の確認なしで push しない。揃ったら：
outputs.json 追記 → commit → push → URL レポート

### 提案の質を高めるコツ

- **過去実績を必ず参照**: 「前回テンプレBで開封35%だった商品Xが反応良かった」など具体的に
- **直近イベントを最初に提示**: ユーザーが状況を思い出すのを助ける
- **3つ以上の選択肢は出しすぎ**: 1メイン提案 + 2 代替で十分
- **ブランドアイデンティティを踏まえる**: NOAHL なら Natural 寄りの提案を厚めに
- **NG表現（激安・格安・量産）は提案にも含めない**

### スキップしていい場面

**件名と配信日時の確認は原則スキップ不可**。
ユーザーが明示的に「件名は任せる、配信日時も任せる」と言った場合のみ、Claude 側で決めて即生成して良い。

テンプレ提案フェーズについては、ユーザーが具体的に指定済みなら省略可：
- ✅ テンプレ提案スキップOK：「品番 abc-1 で **マラソン向けセール告知** 作って」「品番 abc, def, ghi で **テンプレB** で」
- ❌ テンプレ提案スキップNG（提案する）：「品番 abc-1 でメルマガ作って」「メルマガ作って」「品番 abc, def」

ただし上記のいずれの場合も **件名と配信日時は必ず確認** する。

### 生成後の必須レポート項目

メルマガ生成・push が完了したら、**必ず以下を含めて報告する**：

1. **生成したメルマガの ID とタイトル**
2. **Web で確認するための URL** ← 重要、忘れない
   - Production: `https://mail-magazine.vercel.app/outputs/<id>/`
   - Preview（PR 経由の場合）: Vercel が生成した Preview URL
3. 使用したテンプレ・商品・イベント設定の要約
4. 次にユーザーが取るべきアクション（実績入力 / 楽天 RMS にコピペ等）

例：
> 生成完了：
> - ID: `20260510-mothers-day`
> - タイトル: 「母の日ギフト｜やさしい一着」
> - **Web で確認**: https://mail-magazine.vercel.app/outputs/20260510-mothers-day/
> - テンプレ: I（ギフト訴求型）/ 商品: nlwp315-2505 / イベント: 母の日
>
> 配信前にプレビューを確認 → 「HTMLをコピー」で楽天 RMS に貼ってください。

## メルマガ制作フロー

### パターン1: 品番渡される（API 経由生成）

```bash
# Vercel 経由で商品情報取得（認証情報は env vars にある）
curl -s https://mail-magazine.vercel.app/api/rakuten/noahl/<品番>
```

1. 商品情報取得 → JSON で読み取り
2. テンプレ選定（後述「テンプレ選定ロジック」参照）
3. 必要画像のチェック（不足してたらユーザーに確認）
4. テンプレに変数を埋めて HTML 生成（`{{COLOR_*}}` `{{URL_*}}` 等は残す）
5. `frontend/data/brands/<brandId>/outputs.json` に追記
6. commit & push → Vercel 自動デプロイ → Web で `applyBrandToHtml()` が色を置換

### パターン2: 「こういうイベント打ちたい」と相談される

ユーザーから企画概要・商品リスト・画像が共有されたら：

1. **テンプレートを提案**: 商品数・タイミング・訴求ポイントから A/B/C のどれが合うか提案
2. 不足情報があれば確認（画像数、価格、CTA 文言など）
3. 揃ったらパターン1 と同じ流れで生成

### パターン3: 「テンプレB 使いたい」と指定される

1. `templates.json` の B の `requiredImages` / `requiredText` を見て **必要なものを伝える**
2. 揃ったら生成

## テンプレ選定ロジック

| シーン | 推奨テンプレ |
|--------|-------------|
| 新作予約・限定商品の単品プッシュ | A |
| 1〜2 商品の主力訴求 | A |
| 楽天マラソン・SS 開始時の盛り上げ | B |
| 全品クーポン配布告知 | B |
| マラソン終盤・残り24時間の駆け込み | C |
| 「みんなが買った」社会的証明訴求 | C |
| 売れ筋ランキング訴求 | C |

詳細は `templates.json` 各テンプレの `bestFor` / `notRecommendedFor` / `exampleScenario` 参照。

## コンサル役としての役割

メルマガ実装だけでなく、以下の **能動的な提案** を期待されている：

### A. テンプレ提案
既存A/B/C で対応できないシーンには、新テンプレD/E... を提案する。

### B. メルマガ手法・トレンド提案
ユーザーから企画相談がない場面でも、以下のような業界トレンド・成功パターンを能動的に提示する：

**コンテンツ型トレンド**:
- パーソナライズ配信（顧客の過去購入・閲覧に基づく推奨）
- ストーリーテリング（商品の背景・素材の話・スタッフのこだわり）
- UGC 活用（お客様レビュー・着用写真の二次利用）
- スタッフ着用コーディネート（ナチュラル系ECで強い）
- 季節の暮らし提案（読み物型、商品はさりげなく）
- クロスセル（「同素材の別カラーも」「これと相性の良いボトムは」）

**配信戦略**:
- セグメント別配信（年代・購買頻度・ブランド既存ファン度別）
- タイミング最適化（業界平均は火水20-22時、業種で変動）
- 件名 A/B テスト（同内容で件名2案配信）
- リマインド配信（カート放棄24時間後、お気に入り入り3日後）
- カウントダウン型（残り◯時間の緊急性訴求）

**フォーマット型**:
- ミニマル/読み物型（テキスト中心、商品は脇役、ブランド世界観訴求）
- インタラクティブ（クイズ、選べるクーポン、アンケート）
- 連載型（「店長の今週」「素材の話」など継続物）
- ビジュアルカタログ型（写真メイン、テキスト最小）

### C-pre. 「読みたくなるメルマガ」を志向するコンテンツ方針

ユーザーから以下の方向性が示されている：

- **「これは見てみたい」と感じさせる** ことを最優先に。商品カタログ的でなく、エモーショナル・好奇心を刺激する切り口
- **対話式メルマガも面白い**（例: 新入社員視点で「人気商品教えて」「私のおすすめ」「先輩スタッフに聞いてみた」など）
- **過去配信メルマガからデザイン・色合いを参考に** 新テンプレートを提案してOK
- **ブランド統一性は絶対**（色・フォント変更NG。差別化は写真・コピー・構図で）

提案時の問いかけ例：
- 「この商品、ストーリーで売ったほうが反応取れそう。新入社員視点で『初めて触った時の驚き』を語る対話式メルマガはどうですか？」
- 「過去配信のXXXメルマガが反応良かったので、その構成を踏襲して別商品でも作りませんか？」

### C. ブランド・カテゴリ特性に合わせた提案
NOAHL（ナチュラル・カジュアル女性服）に合うもの：
- スタッフ着用コーデ紹介（→ A型派生）
- レビュー・お客様の声まとめ（→ C型派生、社会的証明強化）
- 季節の暮らし＋アイテム提案（読み物型・新テンプレ）
- 母の日・敬老の日のギフト訴求（→ B型派生）
- 再入荷通知（人気商品の入荷待ち客向け）

### D. 過去実績ベースの提案

`outputs.json` の各エントリに `results` がある。これを参照して：

- 「前回テンプレBで開封率35%だった商品Xでもう一度送りましょう」
- 「テンプレAの新作予約は反応が良い傾向」
- 「ランキング型は終盤の駆け込みで売上が伸びる」

など、データに基づいた提案を行う。

## 重要な制約

### 楽天 R-Mail で使える HTML だけ使う（厳守）

メルマガ HTML は楽天 RMS のメルマガ管理画面に貼り付けて配信します。**楽天環境でレンダリングできない記法は絶対に使わない**：

#### ✅ 使ってよいもの
- `<table>` ベースのレイアウト（`width` `cellpadding` `cellspacing` `align` `bgcolor` 属性 OK）
- `<font color size>`（テキストの色とサイズ）
- `<b>` `<strike>` `<br>` `<a target href>`
- `<img width src alt border>`
- インライン `style=""` の **基本プロパティのみ**（`padding` `line-height` `letter-spacing` `border-top` `text-decoration: none`）
- `<!-- コメント -->`

#### ❌ 使ってはいけないもの
- `<style>` ブロック・外部 CSS
- `<script>` ・`<iframe>` ・`<form>` ・`<video>`
- `class=""` （楽天側で剥がされる前提で書かない）
- CSS Grid / Flexbox / `display: flex` / `display: grid`
- `position: absolute` / `position: fixed`
- `border-radius` を多用しない（一部メールクライアントで無視される。装飾用途なら可、ボタン形状などは正方形寄りで設計）
- `background-image` （表示されない端末が多い）
- `@media` クエリ
- Web フォント `@font-face`
- SVG 直書き

#### 判断に迷ったとき
- 既存テンプレ（`brands/<brandId>/templates.json` の A/B/C/D）は楽天で配信実績のある書き方ベース。新規生成時もこのパターンを踏襲する
- 「装飾を凝りたい」場合は CSS ではなく **画像（楽天 cabinet）で表現**する

### 価格訴求時は割引率（%OFF）を必ず入れる

クーポン適用や割引を提案する際は、商品ブロックに **`◯◯%OFF` を必ず明示**する。Web ダッシュボードの「使用商品」セクションは `regularPrice` / `salePrice` から自動計算してバッジ表示されるが、配信用 HTML 側では Claude が生成時に直接書く：

```html
<font color="{{COLOR_MUTED}}" size="1"><strike>5,900円</strike></font>
<font color="{{COLOR_ACCENT}}" size="2"><b> → 2,950円</b></font>
<font color="{{COLOR_ACCENT}}" size="1"><b>（50%OFF）</b></font>
```

色付きバッジ風にしたい場合は、`bgcolor` 付きの 1セル `<table>` を使う（`<span>` の `background-color` は楽天で効かない端末あり）：

```html
<table cellpadding="0" cellspacing="0" align="center" style="display:inline-block;"><tr>
  <td bgcolor="{{COLOR_ACCENT}}" style="padding:3px 8px;"><font color="#ffffff" size="1"><b>50%OFF</b></font></td>
</tr></table>
```

割引率の計算は元値・割引後価格から「(元値 − 割引後) / 元値 × 100」を四捨五入。

### 画像の出所（厳守）

**メルマガに使う画像は必ず楽天 RMS から取得したものを使う。ネット上の画像、プレースホルダー（placehold.jp 等）、フリー素材を勝手に使わない。**

- ✅ **必ず使う**: 楽天 RMS API の `images[]` から取得した cabinet 画像
  - フルURL: `https://image.rakuten.co.jp/<shopUrl>/cabinet<location>`
  - 例: `https://image.rakuten.co.jp/noahl/cabinet/shohin/2025/202505/nlwp315-1r.jpg`
- ✅ **メイン画像（第一画像）**: `images[0].location` を最優先で使用
  - これは商品の代表カット。サムネイルにも使われている
- ✅ **カラー別**: 通常 `<品番>-iv1.jpg`（アイボリー）`<品番>-bk1.jpg`（ブラック）など、カラーコード+番号で並んでいるのでそれぞれの 1 番を使用
- ❌ **絶対NG**: `placehold.jp` `placeholder.com` などダミー画像
- ❌ **絶対NG**: Unsplash・Pexels・Pixabay などフリー素材
- ❌ **絶対NG**: ブランドサイト以外の楽天画像（他店舗の画像流用など）
- ❌ **絶対NG**: AI 生成画像

例外的にバナー画像（セール告知の横長）が RMS に無い場合は、ユーザーに「バナー画像のURLを教えてください」と確認する。勝手に補完しない。

- 柔らかく、ナチュラル

### 件名（subject）作成ルール

NOAHL の過去配信実績から抽出したパターン。新規生成時もこのテイストを踏襲する：

**⚠️ 件名は `title` フィールドに入れる（重要）**
- ダッシュボード一覧表示・楽天 RMS 配信時の件名として使われるのは outputs.json の `title` フィールド
- 別建ての `subject` フィールドは作らない（過去エントリにも無く、表示にも使われない冗長フィールド）
- つまり生成時、`title` には作品名ではなく **配信時の件名そのもの**（`【...】メイン訴求 ＊ サブ訴求` 形式）を入れる

**基本構成**: `【タグ】メイン訴求 ＊ サブ訴求`

**冒頭の【...】タグで「読む理由」を一目で伝える**
- 割引率: `【ALL50%OFF】` `【2日間限定60%OFF】` `【70%OFF】`
- 状態: `【再販】` `【New arrivals】`
- 緊急性: `【ラスト5時間】` `【本日限り】` `【2日間限定】`
- サブブランド/特集: `【mimi by NOAHL】`

**「＊」で2つの訴求を区切る**
- 1通で「メイン訴求 + サブ訴求」を併記すると情報量と読みやすさが両立する
- 例: 「初夏の新作4点に、人気シャツも。＊「今だけ2,990円」で待望の再入荷！」

**具体的な数字を入れる**
- 価格: 「今だけ2,990円」「2点で4,980円」
- 実績: 「5,000枚突破」「5.5万枚突破」「累計1万枚突破」
- 期間: 「今夜21時から」「1週間限定」「2日間限定」

**訴求軸の引き出し**
- 価格訴求（クーポン/タイムセール）
- 新作解禁（New arrivals + 期間限定で割引）
- 再販（人気商品の復活）
- ランキング/枚数突破（社会的証明）
- 季節（初夏・春の主役・春ブラウス 等）

**装飾は控えめに**: `◎` `♪` `♬` `＊` を1つ程度。絵文字は使わない。

**NG表現は件名にも適用**: 激安 / 格安 / 在庫処分 / お買い得 / 半額以下 は使わない。

**実例（過去 NOAHL/mimi by NOAHL 配信実績より）**
- 【ALL50%OFF】初夏の新作4点に、人気シャツも。＊「今だけ2,990円」で待望の再入荷！

### ブランド統一性（最重要）

**メルマガごとに色が変わるとブランド性が損なわれる**ため、絶対に守る：

- ✅ **必ず**: ブランドの色・フォント・フッター URL は `brands/<brandId>/config.json`（ブランド設定）から取得して使う
- ❌ **絶対NG**: メルマガ生成時にカラーコード（#xxxxxx）を直書きする
- ❌ **絶対NG**: アクセントカラーをそのメルマガだけ変える（季節感・キャンペーン感を出すなら別の手段で：見出しテキスト、画像、構図変更）
- ❌ **絶対NG**: ブランドフォントと違うフォント指定
- ✅ **新テンプレ提案時も**: 使う色は必ずブランド config の `colors` 内のものから選ぶ

「春らしさを出したい」「セール感を出したい」場合：
- 色は変えない
- バナー画像・コピー・構図で差別化する
- 例: NOAHL の場合、`primary #8c7b6b`/`accent #4a3b2d` は固定。「春」感は緑系画像や柔らかい商品写真で出す

### 認証情報（重要）

- 認証情報は **Vercel の環境変数** に保存（リポジトリには残さない）

- 環境変数が未設定の場合、API は 500 エラーで `expectedVars` を返してくる（その時はユーザーに Vercel 設定を依頼）
