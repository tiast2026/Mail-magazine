# CLAUDE.md — メルマガ管理システム運用ガイド

このリポジトリは楽天店舗向けメルマガ制作・配信実績管理システムです。
このファイルは Claude Code が新セッション開始時に参照する想定で書かれています。

## システム概要

- **AI 生成役**: Claude Code（このセッション）— 楽天 RMS API で商品情報取得 → テンプレに流し込み → outputs.json に追記 → push
- **Web UI**: Vercel 静的ホスティング — 閲覧・プレビュー・HTML コピー・実績入力
- **データ**: リポジトリ内 JSON で一元管理（push が即 Web 反映）
- API 課金は発生しない設計（生成は CLI 側で完結）

## 主要ファイル

| パス | 役割 |
|------|------|
| `frontend/data/brands.json` | ブランド一覧 |
| `frontend/data/brands/<brandId>/config.json` | ブランド情報（色・ロゴ・固定URL・アイデンティティ） |
| `frontend/data/brands/<brandId>/templates.json` | ブランド別テンプレート（A/B/C/D） |
| `frontend/data/brands/<brandId>/outputs.json` | 生成済みメルマガと配信実績 |
| `frontend/app/api/rakuten/[brandId]/[manageNumber]/route.ts` | 楽天 RMS API ルート |
| `frontend/lib/data.ts` | データ取得 + `applyBrandToHtml()` 色置換ヘルパー |
| `frontend/lib/events.ts` | キャンペーンイベント定義（marathon, supersale 等） |
| `scripts/fetch-rakuten-public.mjs` | 公開ページから OGP 取得（フォールバック） |

## 楽天 RMS API の使い方

**認証情報は Vercel 環境変数で管理**しています。Claude Code は
デプロイ済みの API ルート経由で商品情報を取得します。

### API エンドポイント

```
GET https://mail-magazine.vercel.app/api/rakuten/<brandId>/<品番>
```

例:
```bash
curl -s https://mail-magazine.vercel.app/api/rakuten/noahl/nlwp315-2505
```

レスポンス JSON:
- `manageNumber`: 商品管理番号
- `name`: 商品名
- `tagline`: タグライン
- `url`: 商品ページURL
- `mainImage`: 第一画像 URL（fullUrl）
- `images[]`: 全画像（location, fullUrl, alt）
- `variants`: カラー別 variants（images, standardPrice）

### 環境変数の命名規則

ブランド固有：
- `RAKUTEN_<BRANDID大文字>_SERVICE_SECRET`
- `RAKUTEN_<BRANDID大文字>_LICENSE_KEY`
- `RAKUTEN_<BRANDID大文字>_SHOP_URL`

例: NOAHL なら `RAKUTEN_NOAHL_SERVICE_SECRET`

旧命名（プレフィックスなし）はフォールバック対応。

### メルマガ生成時のフロー

ユーザーから「品番〇〇でメルマガ作って」と依頼されたら：

1. 現在編集中ブランド（通常は default = noahl）を確認
2. `curl https://mail-magazine.vercel.app/api/rakuten/<brandId>/<品番>` で商品情報取得
3. テンプレートを選定（A/B/C/D）
4. テンプレートに変数を埋めて HTML 生成（{{COLOR_*}} 等は applyBrandToHtml で実行時置換）
5. `frontend/data/brands/<brandId>/outputs.json` に追記
6. commit & push → Vercel 自動デプロイ → Web で確認可能

### イベント・配信日付情報

メルマガ生成時に以下も指定する：
- `event.type`: marathon / supersale / blackfriday / yearend / newyear /
  newcollection / preorder / restock / review / regular / custom
- `event.name`: 表示名（例: "2026年5月お買い物マラソン"）
- `event.startDate` / `event.endDate`: イベント期間
- `scheduledAt`: 配信予定日時
- `sentAt`: 実配信日時（後から記録）

ユーザーから「マラソン向けに作って」「スーパーSALE で送る」と言われたら、
適切な `event.type` を設定する。日時はユーザーから聞くか、現在時刻ベースで仮設定。

## セッション開始時の対応（必読）

ユーザーから「メルマガを制作したい」「メルマガ作って」のようなざっくりした
依頼が来たら、**まず以下を確認してから生成に進む**：

### 確認する項目（優先順）

1. **配信先イベント**: どのイベント向けか？
   - 楽天お買い物マラソン / スーパーSALE / ブラックフライデー /
     通常配信 / 新作予告 / 再入荷通知 / その他
2. **配信予定日**: いつ送る予定か？（直近のマラソン日程は今日の日付から推測も可）
3. **商品**: 品番は決まっているか？（複数の場合は何点か）
   - 決まっていなければ、過去実績や提案で代替できる旨を伝える
4. **訴求軸の希望**: ある場合のみ（セール強訴求 / ストーリー / ランキング 等）
   - 未指定なら Claude が商品数・タイミングから最適テンプレを提案
5. **画像確認**: 楽天 RMS で必要画像が揃うか（バナー画像が無い場合のみ要確認）

### 確認の仕方（簡潔に）

「メルマガを制作したい」と言われたら、最初に1メッセージで複数項目をまとめて聞く：

> 何点か確認させてください：
> 1. どんなイベント向けですか？（楽天マラソン / SS / 通常配信 など）
> 2. 配信予定日は？
> 3. 使う商品の品番は？（決まっていれば教えてください、未定なら「過去実績から提案」もできます）
> 4. 訴求軸の希望はありますか？（無ければ私から最適なテンプレを提案します）

最低限 1〜3 が揃えば生成に進める。4 は無くてもOK（Claude が提案）。

### スキップしていい場面

すでに具体的な指示（「品番 abc-1 でマラソン向けセール告知作って」など）が
来ている場合は、確認は省略して即生成に進む。

## メルマガ制作フロー

### パターン1: 品番渡される（API 経由生成）

```bash
# Vercel 経由で商品情報取得（認証情報は env vars にある）
curl -s https://mail-magazine.vercel.app/api/rakuten/noahl/<品番>
```

1. 商品情報取得 → JSON で読み取り
2. テンプレ選定（後述「テンプレ選定ロジック」参照）
3. 必要画像のチェック（不足してたらユーザーに確認）
4. テンプレに変数を埋めて HTML 生成（`{{COLOR_*}}` `{{URL_*}}` 等は残す）
5. `frontend/data/brands/<brandId>/outputs.json` に追記
6. commit & push → Vercel 自動デプロイ → Web で `applyBrandToHtml()` が色を置換

### パターン2: 「こういうイベント打ちたい」と相談される

ユーザーから企画概要・商品リスト・画像が共有されたら：

1. **テンプレートを提案**: 商品数・タイミング・訴求ポイントから A/B/C のどれが合うか提案
2. 不足情報があれば確認（画像数、価格、CTA 文言など）
3. 揃ったらパターン1 と同じ流れで生成

### パターン3: 「テンプレB 使いたい」と指定される

1. `templates.json` の B の `requiredImages` / `requiredText` を見て **必要なものを伝える**
2. 揃ったら生成

## テンプレ選定ロジック

| シーン | 推奨テンプレ |
|--------|-------------|
| 新作予約・限定商品の単品プッシュ | A |
| 1〜2 商品の主力訴求 | A |
| 楽天マラソン・SS 開始時の盛り上げ | B |
| 全品クーポン配布告知 | B |
| マラソン終盤・残り24時間の駆け込み | C |
| 「みんなが買った」社会的証明訴求 | C |
| 売れ筋ランキング訴求 | C |

詳細は `templates.json` 各テンプレの `bestFor` / `notRecommendedFor` / `exampleScenario` 参照。

## コンサル役としての役割

メルマガ実装だけでなく、以下の **能動的な提案** を期待されている：

### A. テンプレ提案
既存A/B/C で対応できないシーンには、新テンプレD/E... を提案する。

### B. メルマガ手法・トレンド提案
ユーザーから企画相談がない場面でも、以下のような業界トレンド・成功パターンを能動的に提示する：

**コンテンツ型トレンド**:
- パーソナライズ配信（顧客の過去購入・閲覧に基づく推奨）
- ストーリーテリング（商品の背景・素材の話・スタッフのこだわり）
- UGC 活用（お客様レビュー・着用写真の二次利用）
- スタッフ着用コーディネート（ナチュラル系ECで強い）
- 季節の暮らし提案（読み物型、商品はさりげなく）
- クロスセル（「同素材の別カラーも」「これと相性の良いボトムは」）

**配信戦略**:
- セグメント別配信（年代・購買頻度・ブランド既存ファン度別）
- タイミング最適化（業界平均は火水20-22時、業種で変動）
- 件名 A/B テスト（同内容で件名2案配信）
- リマインド配信（カート放棄24時間後、お気に入り入り3日後）
- カウントダウン型（残り◯時間の緊急性訴求）

**フォーマット型**:
- ミニマル/読み物型（テキスト中心、商品は脇役、ブランド世界観訴求）
- インタラクティブ（クイズ、選べるクーポン、アンケート）
- 連載型（「店長の今週」「素材の話」など継続物）
- ビジュアルカタログ型（写真メイン、テキスト最小）

### C-pre. 「読みたくなるメルマガ」を志向するコンテンツ方針

ユーザーから以下の方向性が示されている：

- **「これは見てみたい」と感じさせる** ことを最優先に。商品カタログ的でなく、エモーショナル・好奇心を刺激する切り口
- **対話式メルマガも面白い**（例: 新入社員視点で「人気商品教えて」「私のおすすめ」「先輩スタッフに聞いてみた」など）
- **過去配信メルマガからデザイン・色合いを参考に** 新テンプレートを提案してOK
- **ブランド統一性は絶対**（色・フォント変更NG。差別化は写真・コピー・構図で）

提案時の問いかけ例：
- 「この商品、ストーリーで売ったほうが反応取れそう。新入社員視点で『初めて触った時の驚き』を語る対話式メルマガはどうですか？」
- 「過去配信のXXXメルマガが反応良かったので、その構成を踏襲して別商品でも作りませんか？」

### C. ブランド・カテゴリ特性に合わせた提案
NOAHL（ナチュラル・カジュアル女性服）に合うもの：
- スタッフ着用コーデ紹介（→ A型派生）
- レビュー・お客様の声まとめ（→ C型派生、社会的証明強化）
- 季節の暮らし＋アイテム提案（読み物型・新テンプレ）
- 母の日・敬老の日のギフト訴求（→ B型派生）
- 再入荷通知（人気商品の入荷待ち客向け）

### D. 過去実績ベースの提案

`outputs.json` の各エントリに `results` がある。これを参照して：

- 「前回テンプレBで開封率35%だった商品Xでもう一度送りましょう」
- 「テンプレAの新作予約は反応が良い傾向」
- 「ランキング型は終盤の駆け込みで売上が伸びる」

など、データに基づいた提案を行う。

## 重要な制約

### 画像の出所（厳守）

**メルマガに使う画像は必ず楽天 RMS から取得したものを使う。ネット上の画像、プレースホルダー（placehold.jp 等）、フリー素材を勝手に使わない。**

- ✅ **必ず使う**: 楽天 RMS API の `images[]` から取得した cabinet 画像
  - フルURL: `https://image.rakuten.co.jp/<shopUrl>/cabinet<location>`
  - 例: `https://image.rakuten.co.jp/noahl/cabinet/shohin/2025/202505/nlwp315-1r.jpg`
- ✅ **メイン画像（第一画像）**: `images[0].location` を最優先で使用
  - これは商品の代表カット。サムネイルにも使われている
- ✅ **カラー別**: 通常 `<品番>-iv1.jpg`（アイボリー）`<品番>-bk1.jpg`（ブラック）など、カラーコード+番号で並んでいるのでそれぞれの 1 番を使用
- ❌ **絶対NG**: `placehold.jp` `placeholder.com` などダミー画像
- ❌ **絶対NG**: Unsplash・Pexels・Pixabay などフリー素材
- ❌ **絶対NG**: ブランドサイト以外の楽天画像（他店舗の画像流用など）
- ❌ **絶対NG**: AI 生成画像

例外的にバナー画像（セール告知の横長）が RMS に無い場合は、ユーザーに「バナー画像のURLを教えてください」と確認する。勝手に補完しない。

### NOAHL ブランドアイデンティティ（必読）

`brands/noahl/config.json` に正式定義あり。要点は以下：

**Brand Essence**: 日々、幸せを感じられる服をつくる
**Vision**: 年齢や環境にとらわれずに、ファッションを通していつでも自分らしく幸せな日常を過ごせる女性を増やす
**Personality**: ありのままの私を、ひとつ上に導いてくれる人
**DNA**: Being myself（自然体）/ Happiness（幸福感）/ Thoughtful（こだわり・寄り添う）
**Design Principle**: Natural（自然体・柔らかさ）> À la mode（ときめき・おしゃれ）> Honest（誠実・丁寧）
**Image Theme**: 自然体でいられる、ナチュラルな抜け感
**Image Words**: カフェ・ナチュラル・ベージュ・ニュアンスカラー・女性らしさ・抜け感・リラックス・柔らかさ
**ポジション**: MUJI（無印良品）寄りのナチュラルトーン

### ⚠️ NG表現（絶対に使わない）

ブランド定義書で明示的に禁止：

- ❌ **価格売り（激安・格安）** — 「激安」「格安」「驚きの安さ」「お買い得」など価格だけで訴求するワード
- ❌ **量産を感じさせるワード** — 「大量入荷」「お一人様何点まで」「在庫処分」「ファミリーセール」など

割引訴求は必要だが、表現を工夫：
- ✅ 良い: 「期間限定クーポン」「マラソン限定価格」「48時間限定」
- ❌ NG: 「激安」「在庫処分セール」「格安」「お買い得」「半額以下」

### コピーライティングのトーン

- 柔らかく、ナチュラル
- 押し付けがましくない（「絶対買って！」「今すぐ！」のような煽りNG）
- 「あなたらしく」「自然体で」「ちょっとだけ前向きに」のニュアンス
- 親しみやすい一方で、品位を保つ
- カフェのような落ち着いた空気感

### ブランド統一性（最重要）

**メルマガごとに色が変わるとブランド性が損なわれる**ため、絶対に守る：

- ✅ **必ず**: ブランドの色・フォント・フッター URL は `brands/<brandId>/config.json`（ブランド設定）から取得して使う
- ❌ **絶対NG**: メルマガ生成時にカラーコード（#xxxxxx）を直書きする
- ❌ **絶対NG**: アクセントカラーをそのメルマガだけ変える（季節感・キャンペーン感を出すなら別の手段で：見出しテキスト、画像、構図変更）
- ❌ **絶対NG**: ブランドフォントと違うフォント指定
- ✅ **新テンプレ提案時も**: 使う色は必ずブランド config の `colors` 内のものから選ぶ

「春らしさを出したい」「セール感を出したい」場合：
- 色は変えない
- バナー画像・コピー・構図で差別化する
- 例: NOAHL の場合、`primary #8c7b6b`/`accent #4a3b2d` は固定。「春」感は緑系画像や柔らかい商品写真で出す

### 認証情報（重要）

- 認証情報は **Vercel の環境変数** に保存（リポジトリには残さない）
- Claude Code は `https://mail-magazine.vercel.app/api/rakuten/<brandId>/<品番>` を curl して商品取得
- 環境変数が未設定の場合、API は 500 エラーで `expectedVars` を返してくる（その時はユーザーに Vercel 設定を依頼）
- ローカルファイル `data/config.local.json` は廃止済み

### 表記・URL

- 楽天の cabinet 画像 URL は `https://image.rakuten.co.jp/<shopUrl>/cabinet/...` 形式
- フッター URL は必ずブランド config の `fixedUrls` から取得
- HTML 内の改行・特殊文字は楽天 RMS メルマガ管理画面での表示崩れに注意
